# 🌶️ 辣椒智慧防控咨询系统 - 核心工作流与架构设计

本文档详细说明了 PepperGuard AI 系统的底层架构、多模态 RAG 检索链路以及前后端交互工作流。

## 一、 系统整体架构

[cite_start]系统采用前后端分离架构，前端保持极简无依赖，后端依托 FastAPI 提供高性能接口，并接入兼容 OpenAI 格式的大模型 API（如硅基流动、DeepSeek 等）[cite: 43, 44]。

```text
┌─────────────────────┐
│   响应式 Web 前端   │  (HTML/CSS/JS，无依赖，响应式布局)
└──────────┬──────────┘
           │ Fetch API
┌──────────▼──────────┐
│   FastAPI 后端        │  :8001
│  ├── /api/chat       │  ← 核心图文咨询接口（结合 RAG）
│  ├── /api/knowledge  │  ← 知识库检索与管理接口
│  └── /api/upload     │  ← 媒体素材上传接口
└──────────┬──────────┘
           │ OpenAI 兼容格式
┌──────────▼──────────┐
│   大模型 API         │  (LLM 推理与多模态视觉分析)
└─────────────────────┘

核心业务工作流 (RAG & 多模态分析)
当用户在前端发起带有“辣椒病害图片 + 文字描述”的咨询时，系统经历以下完整闭环 ：
Step 1: 多模态数据预处理
前端压缩：用户选择图片后，前端自动将图片压缩至 ≤1MB 并进行 base64 编码，减少网络传输开销 。
后端落盘：通过 POST /api/upload/image 接口将图片保存至后端 uploads/ 目录，供后续审计或二次分析使用 。
Step 2: 轻量级 RAG 知识检索
检索引擎：弃用笨重的向量数据库，系统底层采用 TF-IDF + 余弦相似度算法进行轻量级文本检索，并结合关键词匹配作为兜底策略 。
知识召回：根据用户的文字描述，从包含病毒病、细菌病、真菌病等 5 大类的本地知识库中，精准召回最相关的症状、原因与防治方案 。
Step 3: Context 注入与 Prompt 构建
将召回的结构化知识条目作为 Context（上下文）动态注入到系统 Prompt 中 。
将用户的文本问题与 Base64 图片数据组合成多模态 Message。
Step 4: 大模型推理与结果输出
将构建好的 Prompt 发送给大模型 API 。
AI 结合图像视觉特征与注入的本地农业知识，生成低幻觉、高准确率的拟人化诊断建议，并向前端返回最终结果及引用的知识库来源 。

核心数据模型 (Knowledge Base)
为了支撑精准的 RAG 检索，本地知识库采用了高度结构化的 JSON 数据模型设计 ：
{
  "id": str,
  "title": str,           # 疾病名称，如：辣椒病毒病（花叶病）
  "category": str,        # 所属分类（病毒病/细菌病/真菌病/虫害/营养缺乏）
  "tags": [str],          # 业务标签
  "symptoms": str,        # 症状详细描述 (RAG 检索核心字段)
  "causes": str,          # 发病原因
  "prevention": str,      # 预防措施
  "treatment": str,       # 治疗与用药方案 (RAG 检索核心字段)
  "images": [str],        # 参考图片链接
  "keywords": [str],      # 兜底检索关键词
  "severity": str         # 严重程度: mild/moderate/severe
}
